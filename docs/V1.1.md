# ZipList v1.1 — the lush list (and the cutoff)

v1.0 ships when the device pass + sandbox checkout clear. This doc is
everything we consciously left for 1.1 — and the things we decided ZipList
will never do, so nobody re-litigates them at 2am.

## The headline: pretty share links

`ziplist.app/l/brave-otter-42` instead of `/live/zl_3c08528c-…`. Word
slugs (adjective-animal-number) are the brand move — sayable over a
shoulder, readable in a text. Needs a tiny slug→room store on the Pi
(same JSON-store pattern as the license store). Static shares can get the
same treatment (slug→payload), which also fixes the giant base64 URLs.
Entropy note: ~50M combos + rate-limited lookups + optional room
passwords = fine for unlisted; not for secrets, which ZipList doesn't
hold anyway.

## The house-room answer: portal items

The couple's-house-room ask, solved with one primitive instead of a
workspace system: **a list item that contains a ZipList link renders as a
portal row** — tap it, you're in that list/room. One room stays one list
(the bubble stays comprehensible), but people can weave rooms together
themselves: a "House" list whose three items are Groceries, Chores, and
Holiday — each its own bubble with its own lifespan. Zero new protocol,
zero permissions model, maximum generative output from one variable.
Detection is just "does item text match a ziplist.app link."

## Rooms that live while they're loved

Sliding TTL: activity extends a room's life (touch it weekly, it never
dies; abandon it, it fades on schedule — 24h free / 1yr contributor
today). Server-side change in `party/listRoom.ts` — bundle with the next
PartyKit deploy alongside the queued `sync_rejected` ack from
FABLE-AUDIT.

## Juice and utility, ranked

1. **Copy as text** — plain `☐ item / ☑ item` export in the share flow.
   The export people actually reach for. An afternoon of work.
2. **Hold-the-mascot quick capture** — hold = walkie-talkie one-item add,
   release to commit. Tap-to-record stays the main flow (120s lists need
   no arm cramp); hold becomes the signature bonus move.
3. **Pull-down-to-add** — tug the list top, get a draft row. The one new
   gesture that doesn't fight the horizontal list-swipe. No other
   gestures (swipe-on-row conflicts with list switching; double-tap is
   undiscoverable and taxes single-tap).
4. **Tap presence → who's here** — the avatar cluster expands to names.
   Faces are already DiceBear thumbs (v1.0); this is just the roster.
5. **Voice capture inside live rooms** — guests can only type today;
   the recorder pipeline lives on the home page. Wiring voice-add into
   a room makes the bubble fully ZipList. Medium lift (transcription →
   room snapshot), high payoff.
6. **Section headers in lists** — "— merch —" divider rows, speakable as
   "new section: merch". ONLY if real usage begs for structure. This is
   the 80/20 alternative to nesting; see the cutoff below.

## Housekeeping debt (from the vince/stacey audits)

- Shadow system → ~4 elevation tokens (currently ~9 bespoke treatments;
  hardcoded black/teal outliers in SingleList.css).
- Typography tokens: hero + CTA hand-roll their own clamps; migrate to
  `--font-size-*` so there's one type system.
- `routes/clear-cache` + `routes/contributor/success` still carry
  egg-yolk-era gradient pairs — same flat-CTA sweep as everywhere else.

## The cutoff — ZipList will never:

- **Dates, deadlines, reminders, urgency.** "Later" is a list name, not a
  timestamp. The anxiety machine is the competition, not the roadmap.
- **Nested items / trees.** Lists are the parents, items are the
  children — two layers, separated by a swipe instead of an indent.
  Nesting breaks voice-ticking ambiguity-free capture, and that's the
  product.
- **Multi-list rooms / workspaces.** One bubble = one list = one
  lifespan. Portal items (above) give composition without permissions.
- **Accounts, profiles, image uploads.** A name you can edit and a
  generated face is identity enough. The link is the login.
